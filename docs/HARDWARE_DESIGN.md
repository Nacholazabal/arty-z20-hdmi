# Hardware Design Document - Arty Z7-20 HDMI Subtitle Overlay

**Project:** `arty-z20-hdmi`  
**Board:** Digilent Arty Z7-20 (XC7Z020-1CLG400C)  
**Top-level block design:** `hw/bd/hdmi_in.bd`  
**Top-level wrapper:** `hw/src/hdmi_in_wrapper.vhd`  
**Custom Verilog modules in repo:** `hw/ip/`

## 1. Purpose

This hardware platform captures live HDMI video, buffers frames through PS DDR, applies a subtitle overlay in programmable logic, and retransmits the composited result over HDMI output.

The key thesis-oriented hardware contribution is the move from plain HDMI pass-through to a video pipeline with:

- custom AXI-Stream overlay logic,
- BRAM-backed subtitle bitmap storage,
- PS-accessible control/status registers,
- and a clean separation between video-domain reads and software-domain writes.

## 2. System Overview

### 2.1 Video Path

```text
HDMI IN
  -> dvi2rgb
  -> v_vid_in_axi4s
  -> AXI VDMA S2MM
  -> PS DDR frame buffers
  -> AXI VDMA MM2S
  -> axis_video_overlay_rect
  -> v_axi4s_vid_out
  -> rgb2dvi
  -> HDMI OUT
```

### 2.2 Subtitle Path

```text
PS software
  -> AXI GP0
  -> axi_bram_ctrl_0
  -> subtitle_mask_mem port B

axis_video_overlay_rect
  -> subtitle_mask_mem port A
  -> composited subtitle pixels on outgoing AXI4-Stream video
```

The subtitle mask is stored as a 1 bpp packed bitmap. Software writes the bitmap through AXI, while the overlay IP reads it in the pixel clock domain one word at a time and uses the selected bit to decide whether to output text color, bar color, or passthrough video.

## 3. Custom Hardware Added In This Project

### 3.1 `axis_video_overlay_rect`

Located in [hw/ip/axis_video_overlay_rect/](../hw/ip/axis_video_overlay_rect/).

Responsibilities:

- Accepts AXI4-Stream RGB video in and out.
- Tracks pixel coordinates from `tuser` and `tlast`.
- Draws a configurable subtitle bar.
- Reads packed subtitle bits from BRAM.
- Exposes AXI4-Lite registers for runtime control.
- Provides a sticky SOF status bit so software can update the subtitle mask at frame boundaries.

Tracked control register block:

- `0x00`: `x_start`, `y_start`
- `0x04`: `width`, `height`
- `0x08`: bar color
- `0x0C`: text color
- `0x10`: enable + SOF status

Mapped base address in this design:

- `0x43C30000`

### 3.2 `subtitle_mask_mem`

Located in [hw/ip/subtitle_mask_mem/](../hw/ip/subtitle_mask_mem/).

Responsibilities:

- Implements true dual-port BRAM behavior in Verilog.
- Stores a monochrome subtitle bitmap as 32 pixels per 32-bit word.
- Supports pixel-domain reads on Port A.
- Supports software-side AXI BRAM Controller access on Port B.

Current packaged dimensions:

- width: `256` pixels
- height: `64` pixels
- format: `1 bpp`
- storage footprint: `512 x 32-bit words = 2 KB`

Mapped BRAM window in this design:

- `0x40000000`

## 5. Clocking And Domains

- PS control clock: AXI-Lite register/control domain
- DDR/VDMA memory path: AXI memory domain
- HDMI RX pixel clock: recovered by `dvi2rgb`
- HDMI TX pixel clock: generated by `axi_dynclk`
- Subtitle BRAM Port A: pixel clock domain
- Subtitle BRAM Port B: AXI BRAM Controller / PS control domain

This split is important because subtitle pixels are consumed in the live video domain while software updates happen asynchronously from the processor side.

## 7. Design Decisions

### 7.1 Why keep DDR frame buffering?

The DDR-backed VDMA path decouples input and output timing and leaves room for software-assisted video workflows. It also makes the overlay work easier to validate because the outgoing pipeline is already under controlled timing.

### 7.2 Why use BRAM for subtitle masks?

Subtitle masks are small, sparse, and frequently rewritten. A packed BRAM bitmap is efficient, deterministic, and easy to expose to software through `axi_bram_ctrl`.

### 7.3 Why split overlay control from subtitle storage?

Separating control registers from bitmap memory simplifies both hardware and software:

- overlay registers hold geometry, colors, and enable/status bits,
- BRAM holds only subtitle pixel data,
- and each interface can be updated independently.

### 7.4 Why include a SOF-aware status bit?

Subtitle updates should happen coherently at frame boundaries to avoid visible tearing. The overlay IP therefore exports a sticky SOF indication that software can poll before committing a new bitmap.
